# Функциональность приложения по текущей реализации кода

Документ описывает не целевую спецификацию, а фактически реализованное поведение во frontend-коде на текущий момент.

## 1. Общая картина

Приложение представляет собой клиент дилингового/диспетчерского пульта на `Vue 3 + TypeScript + Pinia`, который умеет:

- авторизовать пользователя через backend;
- поднимать SIP и IM-соединения после входа;
- работать с телефонными сессиями, конференциями, очередью вызовов и завешенными линиями;
- интегрироваться с локальным аппаратным контроллером по WebSocket;
- хранить часть пользовательских настроек и локальных данных в IM preferences и IndexedDB;
- работать как web-приложение и как Electron-приложение.

Основной интерфейс после входа состоит из:

- `Header`;
- центральной рабочей области с роутами;
- `Footer`.

## 2. Авторизация и жизненный цикл приложения

### Вход в систему

Реализован экран логина:

- поля логина и пароля;
- показ/скрытие пароля;
- валидация обязательных полей;
- экранная клавиатура для ввода;
- переключение языка интерфейса прямо на экране логина;
- кнопка с версией приложения;
- выпадающий блок с сетевой информацией и очисткой кэша.

Авторизация идёт через `POST /api/user/login`.

### Сессия пользователя

Реализовано:

- сохранение `accessToken`, `refreshToken` и `userInfo` в `localStorage`;
- восстановление сессии при перезагрузке страницы;
- обновление токена через `POST /api/user/refresh-token`;
- logout с полной очисткой пользовательского состояния;
- смена пароля через `POST /api/user/change-password`.

Маршрутизация защищена:

- публичным является только `/login`;
- остальные маршруты требуют восстановленного пользователя и токенов.

### Инициализация после логина

После входа приложение:

- открывает IndexedDB;
- загружает `who-am-i` и capacity пользователя;
- получает backend-конфигурацию;
- проверяет доступ к микрофону;
- загружает конференции;
- поднимает вкладки контактов;
- инициализирует SIP;
- запускает подписки на статусы;
- загружает историю вызовов;
- загружает обучающие подсказки;
- подтягивает внешние контакты из IndexedDB.

## 3. Главная оболочка и навигация

### Роуты

Есть следующие основные маршруты:

- `/login`;
- `/main/workspace/:workspaceId`;
- `/main/monopoly/contacts`;
- `/main/monopoly/pinnedCalls`;
- `/main/monopoly/groups`;
- `/main/monopoly/callQueue`;
- `/main/monopoly/history`;
- `/main/monopoly/phonebook`;
- `/main/monopoly/settings/...`;
- `/main/monopoly/conference/:conferencePServed`.

Корневой `/main` сейчас по умолчанию редиректит на `Contacts`, а не на workspace.

### Header

В header сейчас находятся:

- переключатели рабочих столов;
- кнопка перехода в настройки рядом с переключателями;
- набор кнопок перехода на monopoly-страницы: контакты, завешенные, группы, очередь, история, телефонная книга.

Особенности текущей реализации:

- в центре header сейчас статичный текст `Call queue`;
- для рабочих столов используется моковый список из 3 workspace;
- выбор workspace и monopoly-страниц сделан через `vue-router`.

### Footer

В footer есть общий каркас:

- брендовый текст;
- зона глобальных контролов;
- пользовательский блок.

Footer присутствует на всех внутренних экранах.

## 4. Текущая реализация рабочих столов

Функциональность рабочих столов реализована частично.

Что есть:

- отдельный маршрут `Workspace`;
- типы `Workspace`, `WorkspaceWidget`, `WorkspaceLayout`, `WorkspacesDto`;
- поддержка layout:
  - `two-equal`;
  - `two-left-narrow`;
  - `two-right-narrow`;
  - `three-equal`;
- реестр типов виджетов workspace:
  - `contacts`;
  - `pinnedCalls`;
  - `groups`;
  - `callQueue`;
  - `history`;
  - `phonebook`;
- отрисовка колонок по layout и вставка виджета по позиции;
- адаптация в одну колонку на узком экране.

Что важно:

- источник данных workspace сейчас не backend, а мок `src/__mocks_/mock-workspaces.ts`;
- в коде зашиты 3 workspace:
  - `workspace-1`: контакты, история, очередь;
  - `workspace-2`: очередь, телефонная книга;
  - `workspace-3`: группы, контакты;
- CRUD рабочих столов, редактирование состава виджетов и сохранение workspace на backend в текущем коде не реализованы;
- если `workspaceId` не найден, показывается экран `Workspace not found`.

Итог: каркас workspace и роутинг уже есть, но полноценная бизнес-модель рабочих столов пока не подключена.

## 5. Контакты

Раздел контактов реализован как самостоятельная monopoly-страница и как workspace-виджет.

Функции:

- отображение карточек контактов;
- отображение карточек конференций в общей ленте;
- отображение линий контакта как отдельных карточек;
- вкладки контактов;
- пагинация;
- серверная загрузка контактов;
- подгрузка и использование внешних контактов из IndexedDB;
- фильтрация состава по вкладке;
- подписка на статусы контактов и линий.

Особенности вкладок:

- создаются системные вкладки конференций и часто вызываемых;
- дополнительно строятся вкладки по серверным группам контактов;
- пользователь может иметь собственные вкладки;
- состав вкладок сохраняется в preferences.

Для frequently called используется локальный счётчик вызовов в IndexedDB.

## 6. Очередь вызовов и call manager

### Очередь вызовов

Есть виджет/страница очереди вызовов:

- показывает активные неприкреплённые к pinned calls сессии;
- умеет подсвечивать активный выбранный элемент;
- открывает нужное состояние call manager по типу сессии:
  - входящий вызов;
  - обычный разговор с контактом;
  - звонок в конференции.

### Call manager

Call manager реализован как набор карточек состояний:

- initial;
- contact view;
- incoming call;
- contact call;
- conference view;
- conference create;
- conference call;
- contact interception.

В приложении есть полноценный store состояния call manager:

- текущий выбранный контакт/конференция;
- текущая выбранная линия;
- привязка к конкретной SIP-сессии;
- синхронизация с active pinned call.

## 7. SIP-сессии и телефония

Реализован слой работы с SIP/WebRTC-сессиями:

- хранение активных сессий;
- разделение очереди и pinned-сессий;
- поиск сессий по `sessionId`, `callId`, `pServed`;
- обработка входящих и исходящих вызовов;
- выбор предпочтительной сессии для абонента;
- автоперевод некоторых сессий в call manager;
- завершение сессий при logout и закрытии вкладки;
- переключение LED-индикации хаба при входящем звонке.

При старте SIP используется backend-конфиг, а WebSocket для SIP принудительно идёт через `/sip`.

## 8. Завешенные линии

Подсистема pinned calls реализована подробно.

Возможности:

- до 16 слотов завешенных линий;
- добавление по `pServed` и по DTO контакта/конференции;
- хранение названия, громкости, состояния микрофона и порядка;
- выбор активной завешенной линии;
- удаление из pinned;
- перенос сессий между pinned-девайсом и обычной очередью;
- глобальный множитель громкости;
- индивидуальное управление микрофоном и громкостью;
- синхронизация состояния микрофона с реальными сессиями;
- сохранение состояния pinned в IM preferences.

Отдельно поддержано:

- закрепление нескольких слотов на один и тот же `pServed` с `slotIndex`;
- работа pinned calls совместно с группами завешенных;
- выбор отдельного устройства для завешенных вызовов.

## 9. Группы завешенных контактов

Раздел групп реализован как отдельный виджет/страница.

Что есть:

- 7 групп;
- для каждой группы есть `micState`, `volumeState`, список участников;
- участниками выступают pinned-слоты;
- режим редактирования групп;
- добавление активного pinned call в группу;
- удаление участника;
- групповое включение/выключение микрофона;
- групповое выключение/включение громкости;
- синхронизация group state с pinned calls;
- сохранение групп в preferences.

В интерфейсе видно заполненные слоты группы, свободные места и карточки контактов/конференций внутри группы.

## 10. Конференции

Поддержка конференций реализована через IM.

Есть:

- загрузка активных конференций из истории/диалогов IM;
- хранение конференций в store;
- страница полноэкранной конференции по `conferencePServed`;
- отображение участников;
- счётчик участников с локализацией;
- выключение/включение микрофона в конференции;
- завершение звонка из conference route;
- создание conference DTO по событиям IM;
- обновление состава конференции по push-событиям;
- удаление участника;
- добавление участника;
- реакция на удаление конференции и исключение пользователя;
- автозапись конференций, если включена в main settings.

## 11. История вызовов

Раздел истории реализован.

Возможности:

- загрузка истории через IM `requestDialogs`;
- фильтрация только call-событий;
- сортировка по времени;
- исключение системных записей;
- обновление истории в реальном времени через `onCallMessage`;
- таблица истории;
- локальная пагинация;
- ручной refresh;
- отображение типа, номера, ФИО, даты/времени, длительности и записи разговора.

## 12. Телефонная книга

Телефонная книга реализована отдельно от карточек контактов.

Есть:

- серверная загрузка списка;
- поиск по строке;
- пагинация;
- таблица телефонной книги;
- повторный запрос при поиске с debounce.

## 13. Настройки

Есть отдельная monopoly-страница настроек с левым меню.

### 13.1 Main Settings

Реализовано:

- сброс прогресса обучающих подсказок;
- выбор языка интерфейса;
- флаг автозаписи групповых вызовов;
- флаг push-to-talk режима;
- флаг звука входящего звонка;
- выбор рингтона входящих;
- регулировка громкости входящего звонка;
- сброс настроек к значениям по умолчанию.

Main settings автоматически сохраняются в preferences с debounce.

### 13.2 Media Devices

Реализовано:

- получение браузерных `audioinput/audiooutput`;
- хранение конфигураций медиоустройств в IndexedDB;
- поддержка пользовательских и controller-managed устройств;
- отдельный pinned device для завешенных вызовов;
- создание нового устройства;
- настройка входа/выхода, иконки, порядка, режимов;
- обработка `devicechange`;
- синхронизация устройств с модулями контроллера;
- статусы устройства `ready/partial/missing`.

Для goose-устройств дополнительно есть:

- `mode`;
- `pttScope`;
- управление клавиатурной подсветкой через контроллер.

### 13.3 Incoming Call Processing

Реализован экран правил переадресации/обработки входящих:

- загрузка услуг абонента;
- загрузка списка forwardings;
- загрузка сценариев;
- загрузка timetable templates;
- отображение таблицы правил;
- удаление правила;
- создание нового правила из шаблона.

### 13.4 Contact Tabs

Реализовано:

- просмотр списка вкладок;
- открытие модального редактора вкладки;
- сохранение набора вкладок в preferences;
- работа с группами контактов для дефолтных вкладок.

### 13.5 Prioritization

Реализовано:

- список кастомизированных контактов;
- добавление контакта через модал выбора;
- хранение собственных рингтонов/цветов/приоритетов по контактам.

### 13.6 Line Keys

Экран привязки кнопок телефонных линий к модулям goose реализован.

Есть:

- список блоков клавиш по найденным goose-устройствам;
- таблица кнопок выбранного блока;
- сохранение привязок в preferences.

### 13.7 Black/White List

Раздел присутствует в маршрутах, но по факту является заглушкой с текстом `Черные/белые списки`.

## 14. Аппаратный контроллер

Есть интеграция с локальным controller backend по `ws://127.0.0.1:8765`.

Реализовано:

- подключение и переподключение;
- регистрация клиента;
- обработка register/message/event пакетов;
- уведомления об ошибках соединения;
- завершение пользовательской сессии при потере контроллера в production;
- отправка команд на подсветку и звук;
- управление hub/goose/handset LED;
- запрос сетевой информации;
- запрос информации о модулях;
- проигрывание тона и мелодии;
- включение подсветки goose-клавиатур.

Контроллер влияет на:

- login flow в production;
- список медиаустройств;
- line keys;
- LED-индикацию входящих;
- режимы работы goose/PTT.

## 15. Локализация, подсказки и уведомления

Реализовано:

- локализация минимум для `ru-RU`, `en-GB`, `zh-CN`, `zh-TW`;
- runtime-переключение языка;
- цепочки обучающих tooltip-сценариев;
- сброс и повторный запуск сценариев;
- системные уведомления об ошибках и действиях.

## 16. Хранение данных

### Через backend/API

Точно используются:

- `/api/user/login`;
- `/api/user/refresh-token`;
- `/api/user/change-password`;
- `/api/user/who-am-i`;
- `/api/user/configuration`.

Отдельные сущности также загружаются через API-слои контактов, переадресации и других доменных модулей.

### Через IM preferences

На сервере через IM preferences сохраняются:

- pinned calls;
- группы завешенных;
- вкладки контактов;
- биндинги кнопок;
- main settings;
- пользовательские рингтоны;
- кастомизация контактов.

### Через IndexedDB

Локально в IndexedDB хранятся:

- внешние контакты;
- счётчики frequently called;
- медиоустройства;
- pinned device.

## 17. Electron и PWA

По коду поддержаны:

- web-режим;
- Electron-режим;
- service worker / PWA;
- отдельные Electron-обработчики сетевых и сертификатных ошибок;
- сборка desktop-приложения через `electron-builder`.

## 18. Что уже реализовано частично или как заглушка

### Реализовано частично

- workspace-система: есть только каркас роутов, layout и моковые данные;
- footer: каркас глобальный, но его бизнес-содержимое по коду не выглядит центральной законченной фичей;
- header: центральная часть сейчас статична;
- black/white list: экран без функционала.

### Не видно в текущем коде реализации

- backend-хранилища workspace-конфигурации;
- CRUD рабочих столов;
- UI добавления/удаления/перенастройки виджетов рабочего стола;
- сохранения workspace на backend;
- реального ограничения workspace до бизнес-правил из кейса через рабочий store.

## 19. Короткий вывод

Текущее приложение уже полноценно реализует основную телефонию, контакты, конференции, историю, работу с устройствами, контроллером, pinned calls, группы завешенных и пользовательские настройки.

Новая workspace-модель в проект заведена, маршруты и виджеты под неё уже есть, но сейчас это именно демонстрационный слой на моковых данных, а не завершённая backend-driven реализация рабочих столов.
